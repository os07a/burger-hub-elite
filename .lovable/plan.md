

# خطة تجهيز نظام برجرهم — مرحلتين

## الوضع الحالي
- 17 صفحة تفاعلية بالكامل ببيانات ثابتة (hardcoded) في الكود
- لا يوجد authentication أو جداول في قاعدة البيانات
- لا يوجد CRUD حقيقي — كل شيء عرض فقط
- `README.md` فارغ، لا يوجد `.env.example`
- المستشار الذكي يعمل عبر Edge Function

---

## المرحلة الأولى: جاهز للنشر والعرض (Demo على الويب)

### 1. إنشاء ملف `.env.example`
```text
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=your-anon-key
VITE_SUPABASE_PROJECT_ID=your-project-id
```

### 2. كتابة `README.md` احترافي
- وصف المشروع وأقسامه الـ17
- التقنيات المستخدمة
- خطوات التشغيل المحلي (`npm install` → `npm run dev`)
- خطوات النشر على Vercel (ربط GitHub → إعداد Build Command: `npm run build` → Output: `dist`)
- متغيرات البيئة المطلوبة

### 3. ملاحظات Vercel
- المشروع SPA — يحتاج ملف `vercel.json` لإعادة التوجيه:
```json
{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }
```

**نتيجة المرحلة الأولى:** نسخة قابلة للنشر فوراً على Vercel كعرض Demo

## المرحلة الثانية: تحويل إلى نظام حقيقي

هذه المرحلة كبيرة جداً وسنقسمها لخطوات متتالية:

### الخطوة 1: إنشاء جداول قاعدة البيانات
إنشاء الجداول التالية عبر migrations:

| الجدول | الوصف |
|--------|-------|
| `profiles` | بيانات المستخدمين (اسم، دور، صورة) |
| `employees` | الطاقم (اسم، وظيفة، راتب، حالة، صورة) |
| `employee_docs` | وثائق الموظفين (إقامة، صحية، عقد) |
| `attendance` | سجل الحضور والانصراف |
| `products` | المنتجات (اسم، سعر، تكلفة، تصنيف) |
| `inventory_items` | أصناف المخزون (كمية، حد أدنى) |
| `suppliers` | الموردون (اسم، تصنيف، بيانات تواصل) |
| `invoices` | فواتير الموردين |
| `daily_sales` | المبيعات اليومية |
| `messages` | الرسائل النصية |
| `user_roles` | صلاحيات المستخدمين (admin/employee) |

### الخطوة 2: إضافة Authentication
- صفحة تسجيل دخول وتسجيل حساب
- دعم Google OAuth
- حماية جميع الصفحات (Protected Routes)
- إنشاء `profiles` تلقائياً عند التسجيل

### الخطوة 3: نظام الصلاحيات
- جدول `user_roles` مع enum: `admin`, `employee`
- دالة `has_role()` (SECURITY DEFINER)
- RLS policies على كل الجداول:
  - **المدير**: قراءة + إضافة + تعديل + حذف
  - **الموظف**: قراءة فقط (بعض الجداول)

### الخطوة 4: ربط الصفحات بقاعدة البيانات
تحويل كل صفحة من بيانات ثابتة إلى:
- `useQuery` للقراءة من Supabase
- `useMutation` للإضافة والتعديل والحذف
- نماذج (Forms) لإدخال البيانات
- أزرار حذف مع تأكيد
- Loading states و Error handling

الصفحات المتأثرة (بالترتيب):
1. **الطاقم** — employees + employee_docs
2. **الحضور** — attendance
3. **الرواتب** — مرتبط بـ employees
4. **المنتجات** — products
5. **المخزون** — inventory_items
6. **الموردون** — suppliers + invoices
7. **لوحة التحكم** — daily_sales (تجميع)
8. **الأرباح** — daily_sales + products
9. **الرسائل** — messages
10. **باقي الصفحات** (الأرشيف، الولاء، السلوك...)

### الخطوة 5: تحديث README والتوثيق النهائي
- تحديث README بخطوات إعداد قاعدة البيانات
- توثيق الصلاحيات والأدوار
- توثيق Edge Functions

## ملاحظات مهمة
- المرحلة الثانية ضخمة — سننفذها على دفعات (كل صفحة أو صفحتين في رسالة)
- سنبدأ بالأساسيات (Auth + جداول + صلاحيات) ثم نربط الصفحات واحدة تلو الأخرى
- `LOVABLE_API_KEY` للمستشار الذكي يعمل داخل Lovable فقط — عند النقل الخارجي يحتاج استبدال بمفتاح Google Gemini مباشر
- بعد إكمال المرحلة الثانية سيكون النظام **جاهزاً للاستعمال الحقيقي اليومي**
