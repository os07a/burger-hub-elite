# خطة: توليد خريطة ذهنية وهيكلة المشروع للمراجعة الخارجية

## الهدف
إنتاج ملفات قابلة للتنزيل تشرح بنية مشروع **برجرهم OS** بشكل كامل، بحيث تقدر تراجعها برة المنصة وتحدد الثغرات والتحسينات.

## المخرجات (3 ملفات في `/mnt/documents/`)

### 1. `burgerhum_mindmap.mmd` — خريطة ذهنية Mermaid
خريطة بصرية تفاعلية تعرض المشروع كشجرة:
- **الجذر:** برجرهم OS
- **6 فروع رئيسية:**
  - الواجهة (17 صفحة مجمّعة حسب الوظيفة: مالية، عمليات، عملاء، موارد بشرية، مخزون، ذكاء)
  - Hooks (28 hook)
  - قاعدة البيانات (28 جدول مجمّعة حسب المجال)
  - Edge Functions (10)
  - التكاملات الخارجية (Loyverse, Twilio, Lovable AI, Meta)
  - قواعد الأعمال (نظام الـ 200 سهم، RBAC، RLS)

### 2. `burgerhum_architecture.md` — وثيقة هيكلة تفصيلية
تشمل:
- **جدول الأقسام:** كل صفحة → Hooks المستخدمة → الجداول → الـ Edge Functions
- **مصفوفة التبعيات:** أي قسم يعتمد على أي بيانات
- **خريطة تدفق البيانات:** من POS / Bonat / المستخدم → Supabase → الواجهة
- **حالة كل قسم:** (يعمل بالكامل / جزئي / Demo data)

### 3. `burgerhum_gaps_review.md` — تقرير ثغرات وتحسينات أوّلي
تحليل سريع مبني على الكود الفعلي:
- **ثغرات أمنية محتملة:** فحص RLS، تخزين Tokens، صلاحيات Admin/Employee
- **ديون تقنية:** تكرار في الـ Hooks، صفحات بدون تحديث realtime، حسابات في Frontend بدل DB
- **فرص تحسين:**
  - أقسام تعتمد على بيانات يدوية (الإيداعات البنكية، المبيعات اليومية) → فرص أتمتة
  - أقسام عندها Demo data ما اتربطت (مثلاً الكاميرات)
  - Edge Functions ممكن تتدمج (مثل send-whatsapp-message + send-whatsapp-reply)
- **تحسينات UX:** بناءً على الأقسام اللي اشتغلنا عليها مؤخراً (توحيد الـ MetricCards، فلاتر الفترات الزمنية)

## التقنية المستخدمة
- استكشاف الكود فعلياً عبر `rg` و `code--view` لقراءة كل صفحة و Hook
- استعلامات على `information_schema` لجلب أعمدة الجداول والعلاقات
- فحص ملفات RLS عبر `pg_policies`
- بناء الخريطة بصيغة `mindmap` (Mermaid v10) عشان تطلع شجرية وأنيقة
- توليد الملفات في `/mnt/documents/` وتقديمها كـ artifacts قابلة للتنزيل

## ما لن أفعله
- لن أعدّل أي كود في المشروع
- لن أضيف ميزات جديدة
- لن أغيّر قاعدة البيانات
- التقرير سيكون **تحليلي ووصفي فقط**، أي إصلاحات فعلية تحتاج موافقة منفصلة بعد المراجعة

## الوقت المتوقع
~5-7 استدعاءات للأدوات لجمع البيانات + توليد الملفات.

## بعد الموافقة
ساحوّل لـ default mode، أجمع تفاصيل أعمق (RLS، schema، hooks)، وأولّد الملفات الثلاثة وأعرضها لك كـ artifacts جاهزة للتنزيل.
